# V2 非功能目标

## 文档目的

首发阶段不仅要把功能做出来，还要提前定义“做到什么程度才算可发布”。这份文档用于给后续 Web、API、CI、部署和质量任务提供统一验收基线。

## 性能目标

- 首发关键页面移动端 Lighthouse Performance >= 90
- 首页与归档页优先首屏可读，避免无必要运行时依赖
- 首发阶段默认优先服务内容可读性，不用为了视觉效果引入重型前端运行时

## 可访问性目标

- 关键页面 Lighthouse Accessibility >= 95
- 交互控件触达尺寸 >= 44px
- 键盘导航、焦点可见性、语义标题层级必须可验证
- 色彩对比需满足基础可访问性要求，不能以视觉风格为由牺牲可读性

## SEO 目标

- 首页、归档页、详情页均具备完整且正确的 meta 信息
- 提供 `robots.txt`、`sitemap.xml`
- 至少提供一类结构化数据输出
- 页面 canonical、标题和描述不应依赖客户端运行时二次生成

## 稳定性目标

- API 必须区分 liveness 与 readiness 所需健康信息
- 每日内容更新失败必须可被监控与告警发现
- 单个来源失败不得静默拖垮整轮内容更新
- 公开接口在首发期优先保证可预测错误，而不是返回不透明异常

## 内容质量目标

- 翻译失败不能静默污染公开页面
- 聚类、排序、翻译异常必须可追踪到来源或任务链路
- 对外展示内容必须保留来源可追踪能力
- 当某类增强信息不可用时，系统应降级到可解释的原始内容，而不是输出混乱状态

## 开发与交付目标

- Web、API、shared-types 的关键变更必须伴随文档更新
- 首发闭环需具备最小自动化测试与 CI 门禁
- 预发或预览环境必须能验证前后端联调，不依赖单机手工判断

## 首发阶段优先级说明

首发阶段优先保证：

1. 内容链路稳定
2. 页面可读、可访问、可索引
3. API 契约清晰、错误可预测

首发阶段暂不追求：

- 极致性能分数
- 复杂个性化推荐
- 高级动效或重交互体验

这些目标只有在不破坏以上优先级的前提下才允许推进。
