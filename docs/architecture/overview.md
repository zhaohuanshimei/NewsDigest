# V2 架构总览

## 目标

News Digest V2 是一次从 V1 清理后重新开始的架构重建。它的首发目标不是把旧实现平移过来，而是建立一个真正前后端分离、API-first、可持续扩展到多客户端的内容系统。

V2 的直接目标有三项：

- 支撑首发 Web 站点通过 API 获取日报、归档、详情和健康状态
- 将内容处理链路沉淀为独立后端能力，而不是构建期读取导出文件
- 为未来 App、小程序、后台运营工具保留稳定的扩展边界

## 模块边界

### `services/api`

后端主服务，负责承载：

- 对外只读 API
- 抓取、规范化、聚类、日报生成、翻译的服务组织
- 调度入口、任务状态、健康检查
- 后续后台能力和运营接口

它是内容系统的唯一业务入口，不直接承担最终展示职责。

### `apps/web`

首发 Web 客户端，负责承载：

- 首页、归档页、详情页、RSS 说明页等对用户可见页面
- SEO、可访问性、阅读体验和视觉系统落地
- 通过 HTTP/API 读取内容资源

它不得直接读取数据库，也不得回退为读取后端本地导出 JSON。

### `packages/shared-types`

共享契约层，负责承载：

- 对外资源模型
- API schema 对应的类型定义
- 前后端共享的数据结构命名

它的职责是保持契约一致，不承载业务逻辑。

### `packages/ui`

共享界面基础层，负责承载：

- 设计 token
- 基础排版与颜色系统
- 可复用的 UI 原语与通用组件

它服务于 Web 首发，也为未来其他前端客户端提供视觉基线。

### `docs/context`

只读背景层，负责保留：

- V1 的业务逻辑总结
- 旧数据库与流程信息
- 部署、测试、工作流、前端优化、翻译与 RSS 的历史结论

它是参考资料，不是实现边界来源。

## 核心数据流

V2 的主链路应统一表达为：

`Source -> Article -> Cluster -> Digest -> API Resource -> Web Client`

其中：

- `Source` 表示新闻来源配置
- `Article` 表示抓取并标准化后的原始内容
- `Cluster` 表示同一事件的聚合结果
- `Digest` 表示某个发布日期对外展示的精选集合
- API Resource 表示对外稳定暴露的只读数据契约
- Web Client 只是消费方，而不是内容生产方

翻译、健康检查、排序、搜索等能力都应围绕这条主链路提供增强，而不是绕开它各自建立孤立输出。

## 分层原则

### API-first

- 前端消费 API，不消费构建期文件。
- 后端以资源接口为第一输出，不以静态导出 JSON 为主要对外形态。

### 前后端独立演进

- Web 页面结构、视觉系统和交互可以独立演进。
- 后端数据模型、任务链路和服务组织也应独立演进。
- 两者通过 `packages/shared-types` 和 API 边界对齐，而不是共享内部实现。

### 内容优先

- 架构决策要服务于“稳定生产日报内容”，而不是优先服务于技术炫技。
- 系统应优先保证内容链路可运行、可观测、可解释。

### V1 经验复用但不盲继承

- V1 已验证的业务链路、数据表、SEO 与 RSS 经验应被提炼吸收。
- V1 的目录结构、构建期耦合与弱前后端分离模式不应直接继承。

## 首发阶段边界

L1 首发阶段只关注：

- 搭建稳定的 Web + API 闭环
- 提供日报、归档、详情、健康检查的首发能力
- 建立可持续迭代的 shared types、UI token 和文档体系

暂不在本阶段解决：

- 完整后台运营系统
- 多客户端 SDK
- 权限体系
- 高级个性化推荐

这些能力在主计划中属于后续层级，不应提前侵入首发架构。
